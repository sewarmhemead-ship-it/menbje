# تقرير فحص البرنامج — Vault AI / POS

تاريخ الفحص: 2026-03-05

## ملخص

تم فحص المشروع (الواجهة، السيرفر، الـ API، والإعدادات). تم إصلاح بعض النقاط وتحسين رسائل الأخطاء. لا توجد أخطاء جذرية تمنع التشغيل.

---

## ما تم إصلاحه خلال الفحص

### 1. توجيه Vercel لصفحات الـ Dashboard
- **المشكلة:** عند الدخول مباشرة إلى `/dashboard/login.html` أو `/dashboard/super-admin.html` كان التوجيه العام `/:path*` يرسل الطلب إلى `/dashboard/index.html` فيعرض الصفحة الرئيسية بدل صفحة الدخول أو السوبر أدمن.
- **الإصلاح:** إضافة قواعد rewrite صريحة في `vercel.json`:
  - `/dashboard/login.html` → `/dashboard/login.html`
  - `/dashboard/super-admin.html` → `/dashboard/super-admin.html`
  - `/dashboard/index.html` → `/dashboard/index.html`
- **الملف:** `vercel.json`

### 2. رسالة خطأ حفظ الفاتورة عند نقص المخزون
- **المشكلة:** عند رفض الـ API للفاتورة بسبب "Insufficient stock" كانت الواجهة تعرض فقط "فشل الحفظ" أو "Insufficient stock" بدون تفاصيل.
- **الإصلاح:** عند وجود `productId`, `available`, `required` في رد الـ API يتم عرض رسالة عربية واضحة:  
  "نقص في المخزون: المطلوب X، المتوفر Y (صنف: ...)"
- **الملف:** `dashboard/index.html` — دالة `doSubmitSalesInvoice`

---

## نقاط تم التحقق منها (بدون أخطاء)

- **تسجيل الدخول:** `/api/auth/login` و `/api/auth/me` يعملان مع JWT والحالة (suspended/expired).
- **إدارة المستخدمين (Super Admin):** استدعاءات `GET/PATCH /api/users` و `PATCH /api/users/:id` للـ industryType و status متوافقة مع الـ API.
- **الفواتير:** إضافة سطر مع كمية أكبر من المتوفر تعرض تحذيراً وتضيف السطر؛ عند الحفظ يتم التحقق من المخزون (واجهة + API) ورفض الحفظ مع رسالة واضحة عند النقص.
- **طباعة تقرير المرتجعات:** استخدام `<\/script>` في السلسلة النصية يمنع خطأ تحليل الـ HTML (تم إصلاحه سابقاً).
- **السيرفر:** تحميل `server/index.js` (ESM) يعمل؛ التفرع بين Vercel و Render صحيح.

---

## توصيات (اختيارية)

1. **كلمات المرور:** في الإنتاج يُفضّل استخدام تشفير (مثل bcrypt) بدل تخزين النص الصريح — التعليق موجود في `server/routes/auth.js`.
2. **البيئة:** التأكد من وجود `.env` وعدم رفع مفاتيح حقيقية (مثل `MASTER_KEY`) إلى المستودع.
3. **السماح بالبيع عند نقص المخزون:** إذا أردت السماح بحفظ الفاتورة حتى مع نقص المخزون (مخزون سالب)، يمكن إضافة خيار في الإعدادات وتعديل التحقق في `POST /api/sales/invoice` وواجهة الحفظ.
4. **Service Worker:** معطّل حالياً لتجنب حلقة إعادة التوجيه؛ يمكن إعادة تفعيله لاحقاً مع سياسة cache مناسبة.

---

## هيكل التشغيل

- **محلي / Render:** `npm start` → Express يخدم الـ API وملفات الـ dashboard، `/` → إعادة توجيه إلى `/login`، `/login` و `/super-admin` يقدمان الصفحات المناسبة.
- **Vercel:** الطلبات تُوجّه عبر `vercel.json`؛ الـ API عبر الدالة في `api/index.js`؛ الصفحات الثابتة من مجلد `dashboard` مع القواعد الجديدة لـ login/super-admin/index.

---

## الخلاصة

البرنامج جاهز للاستخدام. التعديلات المطبقة تحسّن سلوك التوجيه على Vercel ووضوح رسائل الأخطاء عند حفظ الفواتير. لا توجد أخطاء حرجة متبقية من الفحص الحالي.
