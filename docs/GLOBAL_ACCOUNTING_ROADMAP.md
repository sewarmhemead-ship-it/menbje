# خارطة طريق: محاسبة شاملة وعالمية — موجّهة لسوريا

هدف التطوير: تحويل النظام إلى **تطبيق محاسبة شامل وعالمي موجّه لسوريا** — يدعم البيع بالعملات (ل.س، دولار، يورو)، فترات مالية، أصول ثابتة، تسوية بنكية، مراكز تكلفة، ولوحة مؤشرات مالية احترافية.

**توجّه سوريا:** في سوريا لا تُطبّق ضريبة على البيع؛ لذلك **الضريبة (VAT) غير مدرجة** في الخارطة. البيع والفواتير والعملات المتعددة هي المحور، بدون ضريبة.

---

## ما هو موجود حالياً (لا نحتاج إعادة بناءه)

| المجال | الموجود |
|--------|----------|
| **المحاسبة المزدوجة** | دليل حسابات، قيود، إلغاء قيد، ميزان مراجعة، قائمة أرباح وخسائر، تدفقات نقدية، ميزانية |
| **العملة** | SYP كأساس، تخزين USD عند المعاملة، تقرير فرق تقييم العملة |
| **سندات** | إيصال قبض، إيصال صرف، قيد يدوي، تحويل بين حسابات |
| **الدين والذمم** | دفتر مدينين، ربط فواتير البيع بالدين |
| **المبيعات والمخزون** | فواتير بيع، مرتجعات، مخزون، FIFO، كسور وحدات |
| **المشتريات والمصروفات** | فواتير مشتريات، مرتجعات مشتريات، مصروفات (قيود) |
| **التصنيع** | BOM، أوامر إنتاج |
| **المراقبة** | سجل إجراءات، تدقيق |
| **الإعدادات** | علامة تجارية، عملة عرض، ضريبة، تاريخ |

## مرحلة ١ — أساس «عالمي» (عملة أساس + لغة) — بدون ضريبة

### 1.1 عملة أساس قابلة للاختيار + البيع بالعملة (Base Currency & Multi-Currency Invoicing)
- **الفكرة:** إمكانية اختيار العملة الأساسية للمنشأة (ل.س SYP / USD / EUR)؛ الفواتير يمكن أن تكون بالليرة أو بالدولار أو اليورو مع عرض المبلغ والعملة بوضوح.
- **المنطق:** كل المبالغ في القيود والتقارير تُخزَن وتُعرض بالعملة الأساسية؛ المعاملات بالأجنبي تُحول عند الإدخال أو عند التسعير؛ الفاتورة تحمل عملة واضحة (ل.س / دولار / يورو) بدون أي ضريبة.
- **أين:** `server/config/settings.js` → `localization.baseCurrency`، `server/currency/` توسيع؛ فواتير البيع تدعم حقل `currency` و`amountInCurrency`.
- **API:** `GET/PUT /api/settings` (موجود) — إضافة حقل `baseCurrency`؛ تقارير واستدعاءات الفواتير تستخدمه تلقائياً.

*(الضريبة VAT غير مطبقة — التطبيق موجّه لسوريا حيث لا تُطبّق ضريبة على البيع.)*

### 1.2 لغة واجهة التقارير (عربي / إنجليزي)
- **الفكرة:** اختيار لغة عرض التقارير والتصدير (عناوين التقارير، أسماء الحسابات، التسميات).
- **المنطق:** مفتاح في الإعدادات؛ الواجهة أو الـ API ترجع نصوصاً حسب المفتاح (خرائط ترجمة بسيطة).
- **أين:** `server/config/settings.js` → `localization.reportLocale`، ملف `server/i18n/reportLabels.js` (اختياري).

## مرحلة ٢ — فترات مالية وإقفال

### 2.1 السنة المالية والفترات
- **الفكرة:** تعريف سنة مالية (تاريخ بداية/نهاية) وفترات (شهرية أو ربع سنوية)؛ كل قيد يرتبط بفترة.
- **المنطق:** عند الإدخال: تحديد الفترة أو استنتاجها من التاريخ؛ التقارير تُفلتر حسب الفترة أو السنة.
- **أين:** `server/config/settings.js` أو جدول `fiscalYears`؛ `journal.js` يتحقق أن التاريخ ضمن فترة مفتوحة.
- **API:** `GET/POST /api/fiscal-periods`، `GET /api/journal?periodId=...`.

### 2.2 إقفال الفترة / Year-End Close
- **الفكرة:** «قفل» الفترة أو نهاية السنة يمنع التعديل على القيود في تلك الفترة؛ نقل أرباح السنة إلى «أرباح محتجزة».
- **المنطق:** علامة `periodClosed` أو `yearClosed`؛ عند الإقفال: قيد إقفال تلقائي (أرباح → أرباح محتجزة) ثم منع إنشاء/تعديل قيود في الفترة.
- **أين:** `server/accounting/periodClose.js` (جديد)، `server/accounting/journal.js` يتحقق قبل الإدراج.
- **API:** `POST /api/fiscal-periods/:id/close`، `GET /api/fiscal-periods`.

## مرحلة ٣ — أصول ثابتة وإهلاك

### 3.1 الأصول الثابتة (Fixed Assets)
- **الفكرة:** تسجيل أصول (أرض، مباني، آلات، مركبات، أثاث) مع تاريخ الاقتناء، التكلفة، طريقة الإهلاك، والعمر الإنتاجي.
- **المنطق:** حركة عند الشراء: من ح/موردين أو بنك إلى ح/أصول ثابتة؛ شهرياً أو عند الطلب: حساب إهلاك (مصروف إهلاك مدين، مجمع إهلاك دائن).
- **أين:** `server/modules/fixedAssets/` (جديد): `assets.js`، `depreciation.js`؛ ربط مع القيود في `journal.js`.
- **API:** `GET/POST/PUT /api/fixed-assets`، `POST /api/fixed-assets/:id/depreciate`، `GET /api/reports/depreciation`.

---

## مرحلة ٤ — تسوية البنك والمصرف

### 4.1 تسوية البنك (Bank Reconciliation)
- **الفكرة:** مقارنة رصيد الحساب في النظام مع كشف البنك؛ تسجيل بنود الكشف (شيكات، عمولات، فوائد) ومطابقتها مع القيود.
- **المنطق:** رصيد افتتاحي + حركات في النظام = رصيد متوقع؛ المستخدم يطابق مع بنود الكشف حتى يتطابق الرصيد النهائي.
- **أين:** `server/modules/bankReconciliation/` (جديد): قائمة حركات الحساب، بنود كشف، حالة مطابقة.
- **API:** `GET /api/accounts/:id/transactions` (حركات حسب الحساب)، `GET/POST /api/bank-reconciliation?accountId=&statementDate=`.

---

## مرحلة ٥ — مراكز تكلفة ومشاريع

### 5.1 مراكز التكلفة (Cost Centers)
- **الفكرة:** ربط جزء من المصروفات (أو الإيرادات) بمركز تكلفة (قسم، فرع، مشروع) للحصول على تقرير أداء لكل مركز.
- **المنطق:** في القيد أو في الفاتورة: حقل اختياري `costCenterId`؛ تقرير «دخل/مصروف حسب مركز التكلفة».
- **أين:** `store` → `costCenters`؛ `journal` يدعم حقل `costCenterId`؛ تقرير جديد في `statements.js` أو `server/reports/costCenter.js`.
- **API:** `GET/POST /api/cost-centers`، `GET /api/reports/cost-center?costCenterId=&fromDate=&toDate=`.

### 5.2 المشاريع (Projects)
- **الفكرة:** مشاريع ككيانات؛ ربط فواتير ومصروفات وقيود بمشروع؛ تقرير ربح/خسارة لكل مشروع.
- **المنطق:** جدول مشاريع؛ في الفاتورة أو القيد: `projectId` اختياري؛ تجميع إيرادات ومصروفات حسب المشروع.
- **أين:** `store.projects`؛ توسيع `salesInvoices` و `journal` و `expenseRecords`؛ `server/reports/projectProfit.js`.
- **API:** `GET/POST/PUT /api/projects`، `GET /api/reports/project-profit?projectId=&fromDate=&toDate=`.

## مرحلة ٦ — لوحة مؤشرات مالية (KPIs)

### 6.1 مؤشرات جاهزة على الداشبورد
- **الفكرة:** بطاقات أو رسوم: التدفق النقدي، معدل السيولة، عمر ديون المدينين، هامش الربح، نسبة المصروفات للإيرادات، رصيد البنك والحسابات الرئيسية.
- **المنطق:** استدعاء موجود (ميزان مراجعة، قائمة دخل، تدفقات، ذمم) وتجميع أرقام لحساب النسب وعرضها.
- **أين:** `server/routes/api.js` → `GET /api/dashboard/kpis?asOfDate=` يرجع كائن واحد فيه كل المؤشرات؛ الواجهة تعرضها في الصفحة الرئيسية.
- **API:** `GET /api/dashboard/kpis`.

### 6.2 تقارير قابلة للتصدير (PDF / Excel)
- **الفكرة:** تصدير ميزان المراجعة، قائمة الدخل، الميزانية، كشف حساب، تقرير إهلاك كـ PDF أو Excel مع شعار المنشأة (بدون ضريبة).
- **المنطق:** نفس البيانات من الـ API؛ على السيرفر أو الواجهة: توليد PDF (مثلاً مكتبة jsPDF أو puppeteer) أو Excel (ExcelJS موجود في الواجهة).
- **أين:** `server/reports/export.js` أو endpoints مثل `GET /api/reports/trial-balance/export?format=pdf`.

## مرحلة ٧ — متعدد المستأجرين وعالمي (Multi-tenant + GAAP)

### 7.1 مستأجرون حسب الدولة/المنطقة
- **الفكرة:** كل مستأجر (tenant) له إعدادات محلية: عملة أساس، دليل حسابات محلي، سنة مالية (بدون ضريبة في التوجّه السوري).
- **المنطق:** الـ tenant موجود (`tenantId`)؛ الإعدادات والـ CoA تُحمّل حسب الـ tenant.
- **أين:** `server/config/settings.js` يصبح حسب tenant أو يُستبدل بـ `tenantSettings`؛ `chartOfAccounts` قد يكون قالباً حسب دولة.

### 7.2 قوالب دليل حسابات حسب الدولة
- **الفكرة:** قوالب جاهزة لدليل حسابات (سوري، مصري، خليجي، دولي مبسط)؛ عند إنشاء مستأجر أو تفعيل «محاسبة عالمية» يختار القالب.
- **المنطق:** ملفات أو جداول `coaTemplates`؛ عند الاختيار تُنسخ الحسابات إلى `accounts` للمستأجر.
- **أين:** `server/accounting/coaTemplates/` (ملفات أو مصفوفات)، دالة `applyTemplate(tenantId, templateKey)`.

| الأولوية | الميزة | الجهد التقريبي |
|----------|--------|-----------------|
| 1 | عملة أساس قابلة للاختيار + فواتير بالعملة (ل.س/دولار/يورو) | متوسط |

## خلاصة

- **شامل:** إضافة أصول ثابتة، إقفال فترات، تسوية بنك، مراكز تكلفة ومشاريع، ولوحة مؤشرات مع تصدير تقارير.
- **عالمي موجّه لسوريا:** عملة أساس + فواتير بالعملة (ل.س/دولار/يورو) + لغة تقارير + قوالب دليل حسابات؛ **بدون ضريبة** (سوريا لا تُطبّق ضريبة على البيع).

يمكن البدء بمرحلة ١ (عملة أساس + فواتير بالعملة) ومرحلة ٦ (KPIs + تصدير) لأثر سريع، ثم الفترات المالية ثم الأصول والتسوية.
